# 脑控机械臂系统现状与开发建议报告

日期：2026-04-06  
分析范围：`brain_code` 下 MI、SSVEP、机械臂控制、通信、视觉识别相关代码  
目标流程：`MI移动 -> SSVEP确认抓取 -> MI带载移动 -> SSVEP确认放下`

## 一、报告目的

本报告用于系统梳理你目前已经完成的代码基础，明确每个部分已经做到什么程度、距离最终目标还差什么、目前最合理的开发顺序是什么，以及现有代码中哪些部分后续必须优化。

这次结论基于静态代码阅读得出，没有连接 EEG 设备、机械臂、摄像头进行联机实测，因此本报告反映的是“代码层面的完成度”和“系统设计层面的缺口”，不是最终实验效果评估。

## 二、总体结论

从现有代码看，你并不是“还没有系统”，而是已经具备了一个完整系统所需的几乎所有核心模块：

1. 有较完整的 MI 数据采集、训练、实时推理链路。
2. 有可用的 SSVEP 实时刺激与实时解码链路。
3. 有真实机械臂的执行后端，支持移动、抓取、放下。
4. 有一个非常接近最终任务流程的视觉与总控原型。

但是，当前系统还没有真正形成你设想中的统一闭环任务系统。核心问题不是单个模块完全缺失，而是：

1. 还没有一个真正统一的“总控状态机”来把 MI、SSVEP、视觉、机械臂连接成一条任务流程。
2. MI 虽然工程结构已经很完整，但当前模型效果还不足以稳定承担四类在线控制。
3. SSVEP 虽然已经能实时输出命令，但当前实现还是“四目标识别界面”，还没有改造成你需要的“8 Hz / 15 Hz 双选决策框”。
4. 视觉原型已经最接近最终任务，但现在仍然大量使用鼠标和键盘模拟脑控输入。

因此，最优路线不是推倒重写，也不是先单独继续堆 MI 或 SSVEP，而是：

1. 以现有视觉总控原型为基础，建立统一任务状态机。
2. 先把 SSVEP 决策逻辑接入总控。
3. 再补强 MI 数据和在线稳定性。
4. 最后替换掉鼠标控制，完成真正的三阶段脑控闭环。

这说明：

1. 当前模型还不能稳定支持四类 MI 在线控制
2. 左手和舌头类别明显不可靠

MI 模块适合继续作为后续系统的实时解码核心，但目前不适合直接承担最终版四方向闭环机械臂控制。

也就是说：

- 架构可复用
- 接口可复用
- 模型需要继续优化

## 十、当前最优开发步骤

这是最关键的一部分。

如果以实验效率和开发效率为目标，当前最优的顺序应该是下面这样。

### 第一步：先建立统一总控状态机

不要先继续零散改 MI 或零散改 SSVEP。  
先以：

- `05_Vision_Block_Recognition/2026-03_yolo_camera_detection/computer/test2.py`

为基础，重构成真正的总控入口。

建议的新状态定义：

1. `STAGE1_MI_MOVE`
2. `STAGE1_SSVEP_DECISION`
3. `STAGE2_TARGET_SELECTION`
4. `STAGE2_GRAB_CONFIRM`
5. `STAGE2_PICKING`
6. `STAGE3_MI_CARRY`
7. `STAGE3_SSVEP_DECISION`
8. `STAGE3_PLACING`
9. `TASK_FINISHED`

这样做的原因是：

1. 现有 `test2.py` 已经最接近任务流
2. 已经有状态切换框架
3. 已经有机械臂命令发送逻辑
4. 已经有目标与刺激显示逻辑

### 第二步：统一机械臂协议

把：

- `03_RobotArm_Control/2026-03_jetmax_execution_server/test2_robot.py`

定义为唯一协议标准。

也就是说后续所有外部模块都必须只用：

1. `MOVE x y`
2. `PICK x y`
3. `PLACE`

旧的单字符控制和额外角度参数版本都不要再作为主线。

### 第三步：先把 SSVEP 接进总控

原因很简单：

1. SSVEP 当前已经比较接近可用
2. 它更适合做“是否确认”“是否继续”这种低维决策
3. 它比 MI 更快进入任务联调阶段

这一阶段要做的不是继续写四目标大界面，而是拆成两类模式：

1. 目标选择模式
2. 双选决策模式

### 第四步：完成完整任务流程，但暂时保留鼠标移动

也就是说，你可以先做一个“半真实系统”：

1. 目标选择用真实 SSVEP
2. 决策框用真实 SSVEP
3. 抓取与放下用真实机械臂
4. 平面移动暂时仍然用鼠标模拟

这样做的好处是你能先验证：

1. 状态机是不是合理
2. 抓取确认逻辑是不是合理
3. 返回与继续机制是不是合理
4. 机械臂动作链路是不是稳定

### 第五步：集中补强 MI

在任务流程已经跑通之后，再集中提升 MI。

MI 的重点不是“再写更多界面”，而是：

1. 补数据
2. 补 session
3. 看在线稳定输出
4. 看误触发率
5. 看命令延迟
6. 看四类混淆情况

### 第六步：用真实 MI 替换鼠标移动

此时再把：

1. `left_hand -> 左`
2. `right_hand -> 右`
3. `feet -> 后`
4. `tongue -> 前`

正式接入状态机。

### 第七步：增加反馈和异常恢复

最后补：

1. 抓取失败重试
2. 机械臂忙状态
3. 动作超时
4. 返回上一步
5. 中断恢复

## 十一、现有代码最需要优化的地方

### 11.1 MI 部分需要优化的地方

1. 当前四类模型精度不均衡
2. 在线可控性指标还不够明确
3. 应该关注“稳定命令率”和“误触发率”，而不仅仅是离线准确率
4. `left_hand` 和 `tongue` 明显需要重新设计采集与训练策略
5. 后续可能要重新评估窗口长度、阈值、confirmation window 等参数

### 11.2 SSVEP 部分需要优化的地方

1. 不应把 GUI 固定死在四目标模式
2. 应把解码结果封装成更清晰的接口，而不是只写日志
3. 应加入更明确的确认条件，避免误判直接触发动作
4. 需要专门做双选决策 UI，而不是继续复用四方向控制语义

### 11.3 视觉部分需要优化的地方

1. 目标频率不能每帧重排
2. 目标需要有稳定 ID
3. 目标选择和抓取确认应该分开
4. HUD 应明确显示当前阶段、当前目标、当前决策状态
5. 状态切换不能只靠固定延时假设

### 11.4 机械臂控制部分需要优化的地方

1. 应增加动作完成反馈
2. 应增加抓取成功 / 失败反馈
3. 应增加动作日志与时间戳
4. 应增加更稳妥的超时与异常处理

### 11.5 通信部分需要优化的地方

1. 应彻底统一协议
2. 应去掉历史遗留的单字符命令风格
3. 应避免各模块自己私写 socket 格式
4. 最好有统一客户端封装

## 十三、最终判断

从项目基础来看，你已经进入“系统集成阶段”，而不是“从零开始阶段”。

当前最关键的判断是：

1. 机械臂执行后端已经足够成熟，应该直接复用。
2. 视觉总控原型是当前最适合作为主入口继续开发的代码。
3. SSVEP 已经足够接近真实使用，应该优先接入总控。
4. MI 工程结构已经成熟，但模型效果仍然是整个系统成为真实脑控系统的最大风险点之一。

因此，当前最优路线不是继续分散写多个脚本，而是：

1. 先做统一总控状态机
2. 先把真实 SSVEP 接进去
3. 再补强 MI
4. 再完成完整闭环实验

## 十四、一句话结论

你现在最缺的不是新的单独算法，而是一个把现有 MI、SSVEP、视觉、机械臂模块真正串起来的统一任务控制层；而最合理的开发顺序，是先统一状态机和协议，再接入真实 SSVEP，最后补强并接入真实 MI。
